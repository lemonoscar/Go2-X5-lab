# P3 草案

## 目标

`P3` 的目标不是继续做 `rough transfer`，而是进入真正的 `full-open-arm rough locomotion` 阶段：

- rough terrain 保持稳定行走
- 机械臂不再只是 warm-up 小范围动作
- base 与 arm 的耦合关系进入可部署阶段

本草案默认当前机械臂默认姿态采用全零位：

- `arm_joint1 = 0`
- `arm_joint2 = 0`
- `arm_joint3 = 0`
- `arm_joint4 = 0`
- `arm_joint5 = 0`
- `arm_joint6 = 0`

## 为什么 P3 不能直接用标准 Rough 任务

当前 route 训练链路里的 policy 观测接口和标准 `RobotLab-Isaac-Velocity-Rough-Go2-X5-v0` 不一致。

标准 rough 配置中：

- `policy.base_lin_vel = None`

而 `P2a/P2b` route 中保留了 `base_lin_vel`。  
这意味着如果直接切到标准 rough，actor 第一层输入维度会变化，checkpoint 不能无缝续训。

因此 `P3` 应该继续走 route 风格的新任务，而不是直接切标准 rough 任务。

## P3 推荐任务语义

建议新建一个 route 阶段任务，例如：

- `RobotLab-Isaac-Velocity-Rough-Go2-X5-ArmFull-v0`

核心原则：

- 保持和 `P2a/P2b` 相同的 obs/action 维度
- arm command 从 warm-up range 继续放大到 full range
- locomotion reward 向标准 rough 靠拢
- arm reward 进入 fully-open 阶段，但不要一步跳满

## 输入和输出维度是否已经最充分

### 当前 route 阶段的维度

当前 `P2a/P2b` route policy 维度是：

- actor 输入：`259`
- critic 输入：`259`
- action 输出：`18`

actor / critic 输入由以下项组成：

- `base_lin_vel`: `3`
- `base_ang_vel`: `3`
- `projected_gravity`: `3`
- `velocity_commands`: `3`
- `joint_pos`: `18`
- `joint_vel`: `18`
- `actions`: `18`
- `height_scan`: `187`
- `arm_joint_command`: `6`

总计：

- `3 + 3 + 3 + 3 + 18 + 18 + 18 + 187 + 6 = 259`

输出动作维度：

- `12` 条腿关节
- `6` 条机械臂关节
- 总计 `18`

### 这套维度够不够

结论分两部分：

- 输出维度已经足够，不需要再扩
- 输入维度已经够用，但还不是“最充分”

输出为什么足够：

- 对当前控制形式来说，动作就是 `18` 个关节位置目标
- 腿和 arm 都已经纳入动作空间
- 如果没有 gripper 开合或末端笛卡尔控制需求，就没有必要继续扩 action dim

输入为什么只是“够用”，还不算“最充分”：

当前输入已经覆盖了 locomotion 所需的大部分核心信息，但如果 `P3` 的目标是真正的 full-open-arm rough 协调，仍然缺少几类高价值信息：

- 机械臂末端位姿或末端相对 base 的位姿
- 机械臂末端线速度 / 角速度
- 足端接触状态的显式二值信息
- arm / body 接触力或末端负载估计
- 更直接的 terrain slope / 局部地形统计

因此我的判断是：

- `18` 维输出已经足够
- `259` 维输入可以先做 `P3`
- 但如果 `P3` 后期表现出 arm 仍然“只会跟命令，不会协调整机稳定”，优先补输入，而不是补输出

## 我建议的 P3 输入增强

如果你想让 `P3` 更稳，我建议优先增加下面几类观测，而不是改动作维度：

- `eef_pos_rel_to_base`: `3`
- `eef_vel_rel_to_base`: `3`
- `foot_contact_flags`: `4`
- `arm_command_delta_from_default`: `6`

这样输入会从 `259` 变成：

- `259 + 3 + 3 + 4 + 6 = 275`

这不是必须，但这是我认为最值得加的一版。

## P3 命令设计

### Base command

P3 前半段建议先保持和 `P2b` 一样，避免“arm 全开”和“base 命令分布突变”同时发生：

- `rel_standing_envs = 0.15`
- `lin_vel_x = (-0.6, 0.6)`
- `lin_vel_y = (-0.3, 0.3)`
- `ang_vel_z = (-0.8, 0.8)`

等 `P3` 稳定后，再放大到更接近标准 rough：

- `lin_vel_y -> (-0.4, 0.4)`
- `ang_vel_z -> (-0.6, 0.6)` 或保持 `(-0.8, 0.8)` 视训练曲线而定

### Arm command

`P3` 不建议直接一步跳到最终 full range。  
建议做两段 curriculum：

- `P3a`：先把 `P2b` 的 warm-up range 放大到大约 `2x`
- `P3b`：再过渡到完整范围

完整范围建议直接参考 flat 开臂配置：

- `arm_joint1: (-1.2, 1.2)`
- `arm_joint2: (-1.2, 1.2)`
- `arm_joint3: (-1.2, 1.2)`
- `arm_joint4: (-0.8, 0.8)`
- `arm_joint5: (-0.7, 0.7)`
- `arm_joint6: (-0.7, 0.7)`

## P3 奖励函数草案

### 设计思路

`P3` 的 reward 不应该从零开始设计，而应该从 `P2b` 的最终权重继续过渡到 full-open-arm rough 版本。

也就是说：

- 起点：`P2b` 的目标权重
- 终点：`P3` 的目标权重

### 为什么不直接把 `arm_stable_track_bonus` 拉满

当前这个项的 gating 还有问题：

- 它用的是 `norm(target_pos)` 做 gate
- 但 arm command 使用了 `use_default_offset=True`

这会导致默认位姿本身也可能让 gate 变大。  
因此在 `P3` 之前，我建议先修这个函数，再正式把它打开到大权重。

## P3 域随机化建议

P3 建议保留 `P2b` 的质量域随机化，并再补两类：

- 机械臂末端 payload randomization
- mild pushes / external disturbance

建议范围：

- base mass scale: `(0.85, 1.15)`
- other body mass scale: `(0.90, 1.10)`
- base COM: `(-0.03, 0.03)`
- end-effector payload scale: `(0.85, 1.15)`

## P3 PPO 超参数建议

我建议 `P3` 先保持和 `P2b` 同一网络规模，只对训练长度和熵略调：

- `num_steps_per_env = 32`
- `max_iterations = 5000 ~ 8000`
- `save_interval = 250`
- `actor_hidden_dims = [512, 256, 128]`
- `critic_hidden_dims = [512, 256, 128]`
- `activation = "elu"`
- `entropy_coef = 0.002 ~ 0.003`
- `learning_rate = 1.0e-3`
- `num_learning_epochs = 5`
- `num_mini_batches = 4`

## P3 开始前我认为必须调整的点

- 新建 route 风格的 `P3` 任务，不要直接复用标准 rough 任务
- 修 `arm_stable_track_bonus` 的 gate 逻辑
- 给 arm command range 加 curriculum，不要一步到 full range
- 视训练表现决定是否补充 `eef` 相关观测

## 最终判断

如果你现在就要开始 `P3`：

- 输出维度 `18` 已经够了，不用动
- 输入维度 `259` 可以直接开跑，但不是最充分
- 如果只追求能训通，先保持 `259 -> 18`
- 如果追求更强的 base-arm coordination，下一版最值得加的是 `eef` 相关观测，而不是增加动作维度
